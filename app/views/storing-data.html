Storing and sharing data

We need to store all data collected from users so that anyone in MOD can find it easily. Legally, we must store data securely and regularly delete old data.

How to keep research data

When you capture research data, make sure it is:

  • classified and labelled correctly
  • protected by the correct access permissions
  • unable to be changed
  • kept for no longer than 2 years

Check how to classify information in Defence.

Information should be treated as a record as soon as possible after it is created or received.

Avoid only using offline storage, such as a memory stick for your digital records, unless advised to do so by the Defence Information Policy Team. This can make records difficult to access and risks them becoming lost if the technology is no longer compatible in the future.

Do not encrypt digital records.

Do not use self-modifying fields when capturing digital records. For example, a date field that automatically updates to reflect real or current time, will not display correct information when the record is viewed in future.

Anonymise research data

Always anonymise personal identifiable data when storing your research records. This includes things like a personâ€™s name or address. Refer to the participant by code instead.

If needed, create a separate document which matches the codes to the personal identifiable data. The document must be password protected and only shared with colleagues if necessary.

You should retain this information for no longer than 2 years.

Video recordings and transcripts

Anonymising video recordings is time consuming and can be ineffective. Consider whether you really need to store a video or screen recording of a user before your session.

Share research data securely

You must not disclose the identities of people when sharing published statistics and other data gathered in surveys. This includes photographs, videos, and sound recordings, as well as words and numbers.

When sharing transcripts of interviews, you need to anonymise personally identifiable circumstances, especially if dealing with sensitive topics like bullying and harassment.

You must also ensure that it is not possible to identify any individual by cross referencing demographic data items in the dataset.

Before you release any data, you must get a signed Data Access Agreement that explains:

  • what data is needed
  • how the data will be used
  • the full names of everyone who will have access to the data

Any data provided must only be used by the people listed in the Data Access Agreement and for the purpose listed in the agreement.

OFFICIAL data

You do not need to explicitly mark OFFICIAL information.

Your organisational policies will provide detailed guidance but there are some things that you need to remember:

  • only share information with those who have a legitimate need to see it
  • maintain a clear desk and always lock sensitive information away
  • only use authorised IT systems to work with or store information
  • dispose of information appropriately
  • only carry the information that you need when working off site
  • do not discuss sensitive issues in public places
  • report lost or stolen information immediately

You can store OFFICIAL information on MS Teams as long as it is a private, and not public, group setting.

You can store OFFICIAL information on your @digital.mod.uk google drive account, as long as the correct sharing and viewing restrictions are in place.

Documents with personal user information such as telephone numbers and other identifiable data need to be password protected.

OFFICIAL-SENSITIVE data

A small amount of OFFICIAL information is sensitive and should only be shared with people who need to know it.

You can check what type of information is senstive. In these cases, you need to label information as OFFICIAL-SENSITIVE.

You must keep OFFICIAL-SENSITIVE information in LTD libraries or LTD SharePoint sites, which have restricted access.

It's best to use existing LTD libraries and LTD SharePoint sites. If you need to create a new one, contact the iHub team.
